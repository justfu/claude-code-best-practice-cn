# 🔥 Claude Code Ops 4.6 分析。高推理

当 Anthropic 推出像 Opus 4.6 这样的模型时，**模型权重** — 数十亿学习到的参数 — 是冻结的。训练成本极其昂贵（数百万美元，数周的计算）。没有人会在一夜之间重新训练模型。

但权重只是更大系统中的一个层次。研究揭示了至少 **7 种不同的机制** 可以导致真实的或感知到的质量变化，即使模型权重被冻结。

| 问题 | 答案 |
|----------|--------|
| 模型权重在发布后会改变吗？ | **否** — 所有提供商都确认 |
| 模型的行为可能逐日不同吗？ | **是** — 已证明有 ±8-14% 的方差 |
| 是故意的"削弱"吗？ | **否** — 没有故意降级的证据 |
| 基础设施 bug 是真实的吗？ | **是** — Anthropic 确认了 3 个 bug 影响了高达 16% 的请求 |
| 有部分是心理因素吗？ | **是** — 确认偏差和蜜月效应是真实的 |
| 系统提示/后训练会改变吗？ | **是** — 各提供商都有文档记录 |
| 用户应该相信自己的感知吗？ | **部分** — 真实原因存在，但感知放大了它们 |

## 完整的推理栈

模型权重被冻结，但它们之上的**九个层次**可以独立影响你的体验：

```
┌──────────────────────────────────────────────┐
│  你的会话上下文                                │  ← 会话内退化
│  （累积的错误、长对话）                        │
├──────────────────────────────────────────────┤
│  系统提示                                     │  ← 定期更新
│  （安全规则、行为指令）                        │
├──────────────────────────────────────────────┤
│  后训练（RLHF / 微调）                        │  ← 可以悄悄更新
│  （指令遵循、安全对齐）                        │
├──────────────────────────────────────────────┤
│  采样参数                                     │  ← 可以在服务端调优
│  （temperature、top-p、top-k）                │
├──────────────────────────────────────────────┤
│  推测性解码                                   │  ← 草稿模型质量变化
│  （草稿模型预测 + 验证）                      │
├──────────────────────────────────────────────┤
│  MoE 路由 / 批次组成                          │  ← 已证明 ±8-14% 方差
│  （每个请求激活哪些专家）                      │
├──────────────────────────────────────────────┤
│  硬件路由                                     │  ← TPU vs GPU vs Trainium
│  （哪个集群服务你的请求）                      │
├──────────────────────────────────────────────┤
│  量化级别                                     │  ← 可能在负载下变化
│  （FP16 vs INT8 vs INT4 精度）               │
├──────────────────────────────────────────────┤
│  编译器和运行时                                │  ← XLA bug 已证明真实
│  （XLA:TPU、CUDA、硬件特定代码）              │
├──────────────────────────────────────────────┤
│  模型权重（冻结）                              │  ← 这些不会改变
│  （数十亿学习到的参数）                        │
└──────────────────────────────────────────────┘
```

关键心智模型：**冻结的权重 ≠ 冻结的行为**。这就像说"相同的引擎 = 相同的驾驶体验"，而忽略了轮胎、路况、燃油质量和驾驶疲劳。

## 已证实的原因：基础设施 Bug

### Anthropic 2025 年 9 月的复盘

2025 年 9 月，Anthropic 发布了一份详细的复盘报告，揭示了 **三个独立的基础设施 bug** 导致 Claude 在 2025 年 8 月到 9 月期间质量下降。他们的官方声明：

> "我们从不因需求、时间或服务器负载而降低模型质量。用户报告的问题完全是由基础设施 bug 引起的。"

### Bug #1 — 上下文窗口路由错误

Sonnet 4 请求被意外路由到配置为 1M token 上下文窗口的服务器，而不是标准服务器。

- **时间线**：8 月 5 日引入，8 月 29 日负载均衡更改后恶化
- **峰值影响**：最严重时段（8 月 31 日）16% 的 Sonnet 4 请求受影响
- **用户影响**：约 30% 的 Claude Code 用户至少经历了一条降级消息
- **隐蔽细节**：路由是"粘性的" — 一旦你命中了错误的服务器，后续请求会继续发往那里
- **修复**：9 月 4–18 日（跨平台推出）

### Bug #2 — TPU 输出损坏

TPU 服务器上的错误配置导致 token 生成期间出错，将高概率分配给应该很少出现的 token。

- **症状**：英文回答中出现泰语或中文字符，明显的代码语法错误
- **影响**：Opus 4.1 和 Opus 4（8 月 25–28 日），Sonnet 4（8 月 25 日–9 月 2 日）
- **范围**：仅 Claude API；第三方平台不受影响
- **修复**：9 月 2 日回滚

### Bug #3 — XLA:TPU 编译器误编译（最棘手的）

一个修复精度问题的代码更改意外暴露了 Google XLA:TPU 中的**潜伏编译器 bug**。

- **根本原因**：近似 top-k 操作（用于选择最可能的下一个 token）"有时返回完全错误的结果，但仅对某些批次大小和模型配置如此"
- **为何难以发现**：它的行为取决于之前或之后运行了什么操作，以及是否启用了调试工具
- **隐藏数月**：2024 年 12 月的一个先前临时方案一直掩盖着这个更深的 bug
- **影响**：Haiku 3.5 已确认；Sonnet 4 和 Opus 3 的子集疑似
- **解决方案**：从近似 top-k 切换到精确 top-k；接受了"轻微的效率影响"，因为"模型质量不可妥协"

### 为何检测困难

Anthropic 自己的自动化评估没有捕获到用户报告的降级，"部分原因是 Claude 通常能很好地从孤立的错误中恢复。"每个 bug 在不同平台以不同速率产生不同症状，创造了"一种令人困惑的报告混合，不指向任何单一原因。"

关键背景：Claude 运行在**三种不同的硬件平台**（AWS Trainium、NVIDIA GPU、Google TPU）上，每种平台有不同的故障模式、编译器和精度行为。你的请求在不同日子可能命中的硬件不同。

## 已证实的原因：MoE 路由方差

现代大型模型通常使用**混合专家 (MoE)** 架构，其中只有模型参数（"专家"）的一个子集为每个输入激活。一个学习到的路由器决定使用哪些专家。

Scale AI 的研究揭示了一个关键发现：

> "稀疏 MoE 和批量推理的结合产生了不可预测的结果，因为批次的组成可以决定你的查询被路由到哪个专家，而同一批次中其他用户的查询组合不是确定性的。"

### 各提供商测量的逐日方差

| 提供商 | 逐日分数方差 |
|----------|--------------------------|
| OpenAI（GPT-4 变体） | ±10–12% |
| Anthropic（Claude 变体） | ±8–11% |
| Google（Gemini 变体） | ±9–14% |

具体例子：同一模型在**越狱抵抗上一天得分 77%，第二天得分 63%**。相同的模型、相同的权重、相同的测试 — 仅因基础设施就有 14 个百分点的波动。

这意味着即使有零 bug 和零更改，同一模型在不同天也可能产生明显不同质量的输出，纯粹是因为请求的批处理和路由方式。当逐日噪声为 10–15% 时，A/B 测试无法可靠检测 5% 的质量信号。

## 已证实的原因：系统提示和后训练更新

### 系统提示更改

模型权重不会改变，但包裹这些权重的**系统提示**可以随时更新。对 Claude 系统提示演变的分析显示有数十次迭代，"热修复" — 添加来修补不期望行为的短指令 — 被定期添加和移除。

Claude 3.7 的系统提示包含多个针对常见 LLM "陷阱"的热修复指令。Claude 4.0 的系统提示移除了所有这些指令，改为通过后训练中的强化学习来处理这些行为。

### 后训练理论

对于无法解释的质量偏移最合理的理论：公司可以更新**微调和 RLHF**（来自人类反馈的强化学习）而不改变基础模型权重。这在技术上使得说"模型没有改变"是真的，同时仍然通过更新的安全护栏和指令遵循调整来改变行为。

## 已证实的原因：静默模型交换

OpenAI 多次被记录在未经通知的情况下更改用户与之交互的模型：

- 隔夜移除模型选择器，强制用户从 GPT-4o 转到 GPT-5
- 将 GPT-4o 设为需要设置中手动切换的隐藏"传统模型"，没有应用内通知
- 一个"自动切换器"bug 将用户路由到错误的模型
- Plus 订阅者报告模型未经同意切换到"受限版本"

Sam Altman 承认推出过程"比我们希望的更颠簸一些"。Reddit 帖子获得了数千个赞，称新模型是"灾难"和"降级"。

这证明了模型交换在行业中**确实发生** — 有时是故意的（产品决策），有时是意外的（路由 bug）。

## 促成因素

### 负载下的量化

为了经济高效地服务数百万用户，公司可能会提供模型的**量化**版本 — 将精度从 FP16 降低到 INT8 或 INT4。这可以减少 2–4 倍的内存使用并加速推理，但引入了微妙的质量损失。提供商是否在负载下动态切换量化级别存在争议，但技术能力存在并在 vLLM 和 TensorRT 等服务框架中有充分文档记录。

### 推测性解码

现代服务栈使用一个较小的"草稿"模型来预测多个 token，然后由真实模型验证它们。理论上这保留了相同的输出分布，但实际上接受率因领域和上下文而异。开箱即用的草稿模型在某些情况下可能工作正常，但在领域特定任务或非常长的上下文中经常遇到困难。

### 上下文窗口污染

在长时间的编码会话中，早期的错误会在上下文中累积。模型看到自己的错误并可能延续它们。这是单次会话中"Claude 变笨了"的最常见原因 — 不是模型退化，而是上下文污染。

**实用技巧**：当质量感觉不对时使用 `/compact` 或开始新会话。这是你能做的最有实际意义的事情。

## 斯坦福研究 — 以及为何它很复杂

斯坦福和 UC Berkeley（Chen、Zaharia、Zou）2023 年的里程碑研究 — "ChatGPT 的行为如何随时间变化？" — 经常被引用为 LLM 退化的证明。头条发现：

> GPT-4 在"这个数字是质数吗？逐步思考"上的准确率从 2023 年 3 月的 **97.6% 下降到 6 月的 2.4%**。

### 研究证明了什么

- "相同"的 LLM 服务的**行为在短时间内可以大幅改变**
- 不同的能力可能朝相反的方向移动（GPT-4 数学变差，GPT-3.5 数学变好）
- 代码生成质量下降（GPT-4 可执行代码：52% → 10%）
- 该研究创造了 **"LLM 漂移"** 这个术语

### 方法论批评

- 3 月版本使用 **temperature 0.0**，而 6 月版本使用 **temperature 1.0** — 一个增加随机性的根本混淆变量
- 每个任务仅 **500 次查询** — 对于确定的统计声明来说太少
- "数学问题"实际上是是/否问题，模型的猜测模式改变了，而不是数学能力
- 变化可能反映了有意的**后训练安全更新**，而不是退化

该研究证明了一些重要的东西 — LLM 行为随时间改变 — 但机制可能是有意的更新，而不是无意的退化。

## 心理学

### 确认偏差

一旦有人在推特上说"Claude 今天很笨"，你就会开始注意到每一个错误。在没有人抱怨的日子里，你会对相同的错误不以为然。社交媒体放大了这种效应。

### 蜜月效应

用户对新模型有一个初始蜜月期，然后逐渐发现局限性。模型没有改变 — 期望的上升速度快于能力的实际提升。

### 任务难度方差

你的任务每天都在变化。一天难题让人感觉模型变差了，即使它并没有。

### "周末 Claude"神话

尽管许多用户相信按星期几有质量模式，但严格的分析发现**没有一致的证据**支持按星期几的质量模式。一个题为"AI 在周一更笨"的分析结果为空。

### LLM 的随机性

LLM 是概率性的。相同的提示每次都可以产生不同的输出。运气不好的话，你可能会连续得到几个差劲的响应 — 纯粹的随机性，不是退化。

## 结论

用户描述的现象是**真实但归因错误**：

- **正确**：他们的体验在某些天确实变差了
- **错误**：模型被故意"削弱"了

实际原因是以下几点的组合：

1. **基础设施 bug** — Anthropic 的复盘证实（高达 16% 的请求受影响）
2. **MoE 路由方差** — Scale AI 测量的 ±8-14% 质量波动，即使零更改
3. **系统提示和后训练更新** — 各提供商都有文档记录
4. **硬件异质性** — TPU vs GPU vs Trainium，各有不同的故障模式
5. **上下文污染** — 长会话降低会话内质量
6. **确认偏差** — 社交媒体放大感知到的模式
7. **随机方差** — 相同模型、相同提示、每次不同输出

测量问题很严重：±8-14% 的逐日方差意味着你无法将真实的 5% 质量变化与噪声区分开来。这就是为什么"都在你脑子里"和"他们削弱了它"两个阵营都感到自信的原因 — 信噪比使得仅凭个人经验无法判断。

# 🔥 Codex 5.3 高推理和发现

### 报告范围

本节解释为什么用户可能经历一个短窗口期，Claude 输出质量下降，而 Codex 5.3 在编码任务上感觉稳定或更强。重点不在于永久的模型质量排名。重点是在真实服务条件下的短视生产行为。

报告日期：2026 年 3 月 5 日。

### 观察到的模式

报告的模式是：

1. 模型质量在一段时间内可接受。
2. 质量似乎在几天内下降。
3. 质量恢复到接近之前的基线。

这种形状通常是服务栈或推出模式，而不是永久的基础模型能力变化。永久的能力下降通常不会这么快恢复，除非有显式的回滚或修复。

### 高推理：为什么 Codex 5.3 在糟糕的窗口期中可能看起来更好

Codex 5.3 在另一个提供商的降级期间可能明显更强，有几个可能同时发生的技术原因：

1. **产品目标匹配**。Codex 5.3 针对代码生成和 agentic 编码工作流进行了优化，因此即使原始模型实力相同，由于工具编排、仓库推理和以代码为中心的指令调优，也能产生更好的编码结果。
2. **推理策略差异**。提供商独立调优延迟、推理深度和解码默认值。一个提供商更保守的策略在同一天可能看起来比另一个激进的、速度优化的策略"更聪明"。
3. **服务路径分离**。即使两个提供商都托管最先进的模型，它们运行不同的路由层、编译器/运行时栈和推出流水线。一个栈中的事件不意味着另一个有相关的降级。
4. **推出和回滚时机**。如果一个提供商正在推出过程中而另一个是稳定的，用户可以看到大的临时质量分歧，而底层模型权重没有长期的改变。
5. **会话级污染效应**。在长时间的编码聊天中，错误累积会放大感知到的下降。一个竞争的助手可能仅仅因为失败的会话被重置或者其工具循环恢复得更快而感觉更好。

### 详细发现

对于像"Claude 感觉很弱了大约四天，然后回来了"这样的报告，最可能的解释是：

1. 一个提供商端事件、路由问题、解码/运行时 bug 或推出回归影响了部分请求。
2. 问题持续了足够长的时间，在实际工作流中被反复注意到。
3. 问题被修复或回滚。
4. 感知质量快速恢复。

在同一时期，Codex 5.3 可能感觉明显更好，因为它没有共享相同的事件路径，而且编码任务优化放大了实际结果中的差距。

### 此模式的假设排名

| 假设 | 可能性 | 理由 |
|------------|------------|-----------|
| 提供商事件 + 回滚 | 高 | 最匹配多天下降后快速恢复 |
| 服务配置更改（采样/延迟/推理预算） | 高 | 突然行为变化的常见来源，无需模型重训练 |
| 静默别名或快照移动 | 中高 | 可以在没有可见用户操作的情况下改变行为 |
| 仅提示漂移和上下文污染 | 中 | 可以降级会话，但不太可能单独解释广泛的多天报告 |
| 永久基础模型降级 | 低 | 与快速恢复到之前的质量不一致 |

### 什么能确认或反驳此发现

要将此从高置信度推断转变为确凿证据，请收集跨天的相同任务集的请求级遥测：

1. 请求时的精确模型标识符和快照/别名。
2. 提供商暴露的任何后端指纹或发布标记。
3. 解码参数（temperature、top_p、top_k、max tokens）。
4. 延迟、超时和错误率跟踪。
5. 固定编码基准提示集上的结构化质量分数。
6. 失败点的会话长度和 token 上下文深度。

如果质量下降与事件窗口、配置更改或后端指纹转移相关，则事件/配置假设得到确认。如果没有此类转移且降级仅出现在长会话中，则上下文污染成为主要解释。

### 实用工程指导

要减少生产中的逐日方差：

1. 尽可能固定模型快照，而不是使用浮动别名。
2. 存储请求元数据（模型 ID、参数、延迟、错误、响应质量标签）。
3. 为编码任务运行固定的每日金丝雀套件并在回归时告警。
4. 在几次失败后重置或压缩长时间运行的会话。
5. 为事件窗口保留备用提供商/模型路径。
6. 在内部仪表板中将"模型质量"与"服务可靠性"分开。

### 最终结论

Codex 5.3 在短暂的 Claude 降级窗口中看起来更好，这在现代 LLM 运营中是技术合理且预期的结果。最强的解释不是永久的模型崩溃。最强的解释是一个提供商的临时服务路径降级，结合另一个提供商在同一时期的编码特定优化和稳定运营。
