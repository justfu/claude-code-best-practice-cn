## 文字记录

[`0:02`](https://youtu.be/AmdLVWMdjOk?t=2) 模型进步得太快了。如果你在三个月或六个月后问我这个问题，我的回答会完全不同。这是 Boris Cherny。他是 Claude Code 的创始人，也是前 Meta 首席工程师。我们谈到了一切塑造他职业生涯的事情。你能解释一下 latent demand（潜在需求）吗？

[`0:17`](https://youtu.be/AmdLVWMdjOk?t=17) Latent demand 我认为 [音乐] 是产品中最重要的原则。

[`0:22`](https://youtu.be/AmdLVWMdjOk?t=22) 你说过有一些明显的文化差异，而且这很困难。

[`0:26`](https://youtu.be/AmdLVWMdjOk?t=26) 天哪，"困难"这个词太轻描淡写了。那简直是一场噩梦。我们还谈到了 Claude Code 以及 Anthropic 现在正在发生什么。即使 Anthropic 的人数翻了两倍，每位工程师的生产力也因为 Claude Code 增长了将近 70% [音乐]。不要为今天的模型构建，要为六个月后的模型构建。我会推荐给大家的唯一一本对我的工程师生涯影响最大的技术书是——你对与 Codex 和 OpenAI 的竞争有什么看法？我想从你晋升为 Meta 高级工程师的故事开始。那个让你获得晋升的项目背后的故事是什么，你当时在哪里？如果我没记错的话，那个项目是 Chats and Groups，这是一个让 Messenger 和 Facebook 更紧密结合的项目。实际上我在 Meta 最早做的几个项目都跟 Messenger 和 Facebook 有关。第一个项目大概是 Zach 有个想法，要把 Messenger 的聊天和 Facebook 群组同步。大概有几种不同版本的产品尝试把 Messenger 和 Facebook 拉近。背后的动机是有种感觉——这种公共空间社交产品正在消失，人们更多地转向聊天和更随意的实时空间。所以我们尝试了几种版本的产品，Chats and Groups 是其中成功的那个。我觉得它可能是第三个或第四个版本。我当时在 Facebook 群组，跟 Messenger 在组织上离得很远。Steve 当时是产品经理，他有了这个想法——我们应该构建这个东西。我就接了过来，说好的，我们干吧。然后我开始动手。很快就有了一些好迹象。所以我申请了更多工程师，有三个人加入了——Shatambri、Crystal 和 Chaang，他们是第一批加入的工程师。后来还得到了数据科学和设计的支持。一开始只在 Web 上，后来也移到了移动端。我觉得我们证明了在 Facebook 群组中可以有聊天功能，这种产品是可行的。里面有很多东西其实完全不工作。以现代产品标准来看，体验超级粗糙。但那时候所有人都在 Web 上构建，各种 bug 都是可以接受的。产品做大了，我们团队这么小，每个人都得什么都干。我们没有用户研究员，所以我会在午餐时间去食堂，有新功能就给食堂工作人员看。嘿，你能弄清楚怎么打开聊天吗？有时候他们能找到，有时候找不到。这就像是一种观察式用户研究。你会看到人们在特定情境下完成任务的方式，你不会给太多提示，然后你就能看到他们卡在哪里。我们就是这么做的，然后我教团队也这么做。很快我们都会在午餐时间去食堂，找食堂工作人员当代表用户来验证设计是否合理。

你觉得早期 Facebook 的文化让工程师在写代码之外也能做很多事情很有意思。你提到你也做了用户研究和设计，甚至在指导别人做设计。这其实是 Facebook 文化中比较独特的一点。我觉得这非常重要，而且至今如此。在 Claude Code 团队我们非常重视通才。我喜欢跟通才一起工作。如果你是个工程师，能写代码，还能做产品、设计、有产品感、喜欢跟用户交流——我喜欢这种工程师。我们现在招聘所有岗位都是这样。我们的产品经理也写代码，数据科学家也写代码，用户研究员也写一点。我就是喜欢这些通才。我从 18 岁做第一个创业项目开始，每个岗位我都得自己干。到 Facebook 之前，我一直在小公司工作，什么都得自己来。在大公司你会被推入某个特定的泳道，但这只是官方说法。工程其实是一个很窄的技能集，但你真正做的是构建产品或基础设施，从头到尾还有很多其他工作不仅仅是写代码。

那段时间产品成功的背后有一个叫 latent demand 的概念。能解释一下吗？Latent demand 我认为是产品中最重要的原则。如果你看看 Facebook 的成功产品，几乎每一个都有 latent demand 的元素。比如 Marketplace，来自一个观察——Facebook 群组中 40% 的帖子是买卖东西。Facebook 群组不是为商业设计的，但人们就这样在用。你设计一个产品让它可以被 hack、被滥用，然后看数据看人们怎么用，然后围绕它构建产品。Facebook 群组里有买卖群组，这成功了，因为人们本来就想在 Facebook 群组上买卖。Marketplace 是自然的延伸。Facebook Dating 也类似。60% 的个人资料浏览者是异性且不是好友。产品原则是——你永远无法让人们做他们还没有在做的事。你能做的是找到他们已有的意图，然后帮助他们更容易地实现它。

你还提到你跨组织工作，连接 Messenger 和群组的工程。有什么建议？天哪，"困难"这个词太轻了。那简直是噩梦。Facebook 当时想快速发布产品，但 Messenger 只关心可靠性和性能。完全相反的价值观。不仅仅是文化差异，Messenger 的工程师甚至怀疑我们，因为我们会影响他们的性能指标。组织目标也完全不同。他们有 SLA 和正常运行时间目标，我们只关心日活用户和参与度。

如果你能重来，你会怎么改变？我可能会去找 Zuck，说如果真想做成这件事，应该把 Messenger 移到 Facebook 组织下。共同的汇报线不能是 Chris Cox 这种级别的，必须更低一些，这样组织结构才能更协作。

你职业生涯中有一些很有趣的副项目。比如 Undux，React 的状态管理框架。Undux 来源于 React 状态管理不必要地复杂。当时最好的方案是 Redux，但我就是搞不懂。所以我构建了一个更简单的替代方案。我在一个非营利组织做志愿者时开始使用，他们的工程师很喜欢。在 Facebook 内部发布后，我发现很多工程师对 Redux 也有同样的挫败感。我还写过一本 TypeScript 的书，因为当时没有好的学习资源。我还创办了当时世界上最大的 TypeScript meetup。

你在 Instagram 期间做了从 Python 到 Hack 的迁移。这个项目有多大规模？从 Facebook 群组开始大概 12 个工程师，最终可能有 20 到 30 个工程师，大约两年。用 Claude Code 的话，现在可能只需要 5 个工程师 6 个月。

你对来到 Anthropic 的决定是什么？我记得第一次用 ChatGPT，当时我在日本乡下，是镇上唯一的工程师，唯一说英语的人。每天早上读 Hacker News，用 ChatGPT 时被震撼了。我在 Anthropic 的第一顿午餐跟 Ben Mann 一起，我提到一本很冷门的科幻小说，桌上所有人都读过。这帮人都是疯狂的科幻迷，而且对安全问题非常认真。我知道这些技术如果走偏会有多糟。

你现在怎么看 Claude Code 的竞争？我其实不太用其他产品。我告诉团队的是，不要被竞争对手分心。我们专注于解决我们自己的问题和用户的问题。

最后一个问题——如果你能回到刚入行时，会给自己什么建议？运用常识。在大公司有很多东西会把你从常识中拉开。信任自己，培养你的常识。
